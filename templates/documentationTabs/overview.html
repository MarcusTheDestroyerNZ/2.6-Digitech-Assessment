Project Overview

A simple game about helping the environment, made for the AS91896 (2.7) assessment.

Version 1.0

Structure and Reuse

I used functions to avoid repeating the same code. For example, power, research, and drawing on the screen are each handled in their own parts.

Key things like research_upgrades() and building_stats() made the game easier to manage and grow.

The game's code is built in layers. The main loop uses other functions to handle everything, like update_power() managing timers, repairs, and limits all at once. This keeps things neat and makes it easier to change or fix stuff.

Game logic (how things work) is separate from UI (how things look), which makes it easier to test and update parts without breaking others.

Design Choices

  • Clicker-style game makes it fun and addictive
  • Environment theme helps match the assessment goals
  • Game gets harder over time to keep it interesting

Technical Features

  • Used dictionaries to handle upgrades, timers, and resources easily.
  • Built a research tree with costs, unlocks, and visuals that connect upgrades.
  • Saved and loaded game progress with JSON to keep track of everything.
  • Split code into different files for things like messages, locking, and UI to stay organised.
  • Used functions like update_power() and idle_reward() to keep the code tidy and reusable.

Reflection and Problem Solving

While building the game, I ran into some issues and fixed them by testing and trying different things:

  • Offline rewards: I made sure the game couldn't be tricked by changing the clock. If it sees a future time, it resets the reward timer.
  • Research tree: It was hard to keep the lines between upgrades in the right place, so I used special data setups and zoom scaling to fix it.
  • Broken save files: Sometimes the save was empty or half-done. I used try/except to stop it crashing and start a new save if needed.
  • Balancing: Some upgrades were way too strong. I wrote a testing script to see how much they gave over time, then fixed the numbers.

Through all this, I learned how useful it is to reuse code, make backups, and test things in small chunks before adding them to the main project.
